# 수업앱 빌더 — 설계 문서

교사가 코딩 없이 학생용 앱(제출·설문·평가)을 만들고, 학생 제출을 모아 피드백·생활기록부까지
잇는 도구. jinro-community 안에 통합(별도 Firebase 안 만들고 기존 것 재사용).

> 이 문서는 **미구현/설계 확정** 파트를 담는다. 구현된 부분은 §0 요약 참조.
> 대화(2026-07-04)에서 확정된 설계 결정을 구현 가능한 수준으로 정리.

---

## 0. 현재 상태 (구현됨, 미검증 — WSL 실행 필요)

**중앙 모드** (우리 Firebase, `builder_*` 컬렉션):

- **수업앱 = 블록 모델**: 앱은 순서 있는 **블록 목록** — 📄제시(자료: 텍스트/이미지/링크) + ✍️입력(문항: 단답/장문/숫자/객관식) + 🤖AI(문항 종속). "자료 제시 → 활동"이 한 앱에. `blocks` 저장 + `fields`(입력만) 파생. 공유 URL `/a/{code}`. 문항 자동번호.
- **문항 종속 AI 블록**: 문항 헤더의 "🤖 AI 기능 추가" → 그 문항의 답을 읽어 피드백. 세기 간단(Haiku)/정밀(Opus), 학생공개 토글. `inputFieldIds`로 문항 참조.
- **제출·대시보드**: 학생 무로그인(이름), 교사 대시보드에서 제출·AI결과 실시간.
- **생활기록부 2단계**: 칸(세그먼트) 설계(교사칸=태도·평가 / AI칸=활동요약) → ① AI칸 자동채움 → ② 교사가 교사칸 입력 → ③ AI가 한 문단으로 엮기 → CSV. 모델 Haiku(무료).
- **연도·학기**: 앱에 `year`·`semester`. 생기부 선택 시 필터.
- **명렬(반) 관리 step1**: `builder_rosters`(반이름·학교·학생[학번,이름]), 붙여넣기 명단·편집.

**과금 라인**: Haiku(요약·형성피드백·생기부) = 무료 / Opus(정밀) = 유료.

**핵심 파일**: `src/lib/builder/{types,repository,rosterRepository,auth,ai,anthropic,bytes,records}.ts`,
`src/app/api/builder/{apps,apps/[id],submit,records,records/compose,rosters,rosters/[id]}`,
`src/app/{builder,builder/[id],builder/records,builder/rosters,a/[code]}`,
`src/components/builder/{AppBuilder,StudentForm,AppDashboard,RecordMaker,RosterManager}.tsx`.

---

> ⚠️ **파일 업로드(이미지/PDF) 설정 — Firebase Storage 필요**. 제시 블록의 파일 업로드는
> `firebase/storage` 클라이언트로 `builder_uploads/{uid}/`에 올리고 downloadURL을 블록에 저장.
> 사전 준비: (1) Firebase 콘솔에서 **Storage 활성화**(기본 버킷 생성), (2) `.env.local`의
> `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET` 설정, (3) **Storage 규칙**:
> ```
> rules_version = '2';
> service firebase.storage {
>   match /b/{bucket}/o {
>     match /builder_uploads/{uid}/{allFile=**} {
>       allow read: if true;                        // 학생이 자료 열람
>       allow write: if request.auth != null
>                    && request.auth.uid == uid
>                    && request.resource.size < 20 * 1024 * 1024;
>     }
>   }
> }
> ```
> 미설정 시 업로드가 실패하고 "URL 직접 붙여넣기"로 폴백됨(유튜브·외부 이미지 URL은 설정 없이 동작).

## 1. 저장 2모드

| | ① 중앙 모드 (구현됨) | ② 데이터 주권 모드 (설계) |
|---|---|---|
| 데이터 위치 | 우리 Firebase | **교사 본인 구글(시트)만** |
| 3자 경유 | 있음(우리 서버) | **없음** |
| 설치 | 링크만 (0) | 교사가 배포 1회 |
| AI | 우리 처리(Haiku 무료/Opus 유료) | **교사 본인 API 키** |
| 대시보드·생기부·명렬 | 매끄러움 | 교사 워크북 안에서 자체 완결 |
| 대상 | 대부분 교사 | 개인정보 엄격 학교·교사 |

모드는 **앱(또는 교사) 단위 선택**. 중앙이 기본, 주권은 옵션.

## 2. 데이터 주권 모드 (Apps Script) — 설계

### 2.1 원리

진짜 "3자 미제공"의 유일한 경로 = **교사 본인 Apps Script가 본인 시트에 기록**.
데이터 흐름: `학생 → 교사의 Apps Script(웹앱) → 교사의 시트`. 우리 서버를 **안 거침**.

부수 효과: 중앙 OAuth 100명 심사벽 없음 — 각 교사가 *자기 스크립트를* 승인하므로,
우리가 심사받을 중앙 OAuth 앱이 없다. 마찰은 "심사"가 아니라 "교사가 1회 배포"로 바뀐다.

### 2.2 자기 완결형 워크북

주권 앱 = **하나의 구글 스프레드시트 워크북 + bound Apps Script**. 탭 구성:

- `제출_{과제}` : 과제별 학생 제출 (열 = 문항 + 이름/학번 + 타임스탬프 + AI결과 + 교사첨언)
- `명렬` : 반 학생(학번·이름)
- `설정` : (숨김 권장) 없음 — 키는 시트가 아니라 Script Properties에 (§2.5)

대시보드·생기부·명렬이 **모두 이 워크북 안에서** 돌아 자체 완결. 우리 중앙 기능을 안 씀.

### 2.3 빌더 → 생성물

중앙 빌더에서 앱 config를 그대로 설계 → **Apps Script 코드 + 시트 구조 자동 생성**:

- `fields` → 학생 폼 HTML(`doGet`) + 제출 탭의 열.
- `doPost` → 제출을 시트에 append.
- `aiBlocks` → 제출 시 `UrlFetchApp.fetch(anthropic)`로 교사 키 사용해 피드백 생성 → 시트에 기록.
- 생기부 → 시트 커스텀 메뉴 버튼(스크립트가 탭들 읽어 Anthropic 호출 → 생기부 열/시트에 출력).

교사 배포 흐름:
```
빌더에서 "주권 모드로 내보내기" → [템플릿 워크북 사본 만들기] 링크(코드 내장)
 → 교사 드라이브에 사본 → 웹앱 배포(권한 승인 1회) → 본인 exec URL
 → 설정 화면에서 본인 API 키 입력(§2.5)
```
"코드 자동생성 → 사본/복붙 → 배포"(대화 초반 아이디어)가 이 모드 전용 경로.

### 2.4 기능 대응 (중앙 vs 주권)

| 기능 | 중앙 | 주권(워크북) |
|---|---|---|
| 폼·제출 | Firebase | 시트 탭 |
| 문항 AI 피드백 | 우리 키 | 교사 키(UrlFetch) |
| 대시보드 | 웹 | 시트 그 자체 |
| 명렬 | 컬렉션 | `명렬` 탭 |
| 생기부 | 2단계 UI | 시트 메뉴 버튼(탭 종합) |
| 학생 열람 | (명렬 로그인) | 워크북 배포 URL에서 학번 조회 |

핵심: **주권 모드는 워크북 하나로 자체 완결**되게 설계하면 중앙 기능을 거의 재현 가능.
단, 실시간 반 모니터링(여러 교사 통합 뷰)은 본질적으로 중앙 전용.

### 2.5 API 키 처리 (교사 본인 키)

**물리적 사실**: "우리가 데이터 안 봄" + "우리가 AI 돌림"은 동시 불가.
→ 주권 모드 AI = 교사 본인 키(데이터가 우리 아닌 Anthropic으로 직행). 키·요금은 교사 소유.

**저장 위치 = Apps Script Script Properties** (`PropertiesService.getScriptProperties()`):
코드에 안 박히고, 시트에도 안 보이고, 우리한테도 안 옴.

**입력 방법** (생성 코드에 내장):

| 방법 | 친절 | 보안 | 채택 |
|---|---|---|---|
| ① 설정 화면 내장(코드가 Script Properties에 저장) | ○ | ○ | **채택** |
| ② Apps Script 편집기 직접 | ✕ | ○ | 대체 |
| ③ 시트 셀 | ○ | **✕(공유 시 유출)** | **금지** |

생성 코드에 **"설정" 웹 화면 + [키 확인] 버튼** 내장 → 교사가 붙여넣기 1회.
③(시트 셀)은 실수 유출 최다 경로 → 막고 경고.

> 중앙 모드에서 "내 키로 쓰기(BYO key)"는 우리가 키를 잠깐이라도 쥐어야 해 주권과 성격이 다름 → 별도 판단(기본 미지원).

### 2.6 한계·리스크

- **배포 마찰**: 교사가 웹앱 배포(권한 승인 화면). 앱당 1회. → 3컷 가이드 + 설정화면으로 완화.
- **Apps Script 쿼터**: 실행시간·UrlFetch 일일 한도, 시트 동시쓰기 잠금(반 동시 제출 시 지연). 대규모 반엔 주의.
- **키·요금 교사 책임**: 회전·정지 우리가 못 함. 키 확인 버튼 필수.
- **버전 업데이트**: 우리가 코드 고쳐도 교사 사본엔 자동 반영 안 됨 → "새 버전 받기" 안내 필요.
- **별도 스택**: 생성기는 Apps Script(JS) 문자열 생성 — Next/Firebase와 다른 트랙.

## 3. 남은 중앙 파이프라인 (명렬 배선 → 리포트)

목표 흐름: 과제(연도·학기) → 학생 제출 → 과제별 요약/피드백 → 교사 첨언 → 학생 열람 →
과제별 "요약본" 누적 → 교사가 연도/학기로 활동 선택 → 버튼 하나로 생기부(요약본 종합).

### 3.1 앱 ↔ 명렬 연결
- `app.rosterId?: string`. 빌더에서 앱 만들 때 명렬(반) 선택(선택).

### 3.2 학생 식별
- `/a/{code}`에서 앱에 rosterId 있으면 → 이름 입력 대신 **명단에서 자기(학번·이름) 선택**.
- `submission.studentNo` 저장. rosterId 없으면 기존 이름 방식 유지(하위호환).

### 3.3 과제별 요약본 저장
- `submission.summary?: string` — 제출 시(또는 교사 트리거) 생성된 과제별 AI 요약/피드백을 **저장**(지금은 즉석 표시만).

### 3.4 교사 첨언
- `submission.teacherNote?: string` — 대시보드에서 학생별 제출에 코멘트. 저장 라우트(owner 검증).

### 3.5 학생 열람
- 명렬 로그인 기반: 학생이 `/a/{code}`에서 자기 학번 선택 → 본인 최근 제출의 **요약+첨언** 조회.
- (경량 인증: 학번 선택. 필요 시 PIN/QR 추후 — [[사고뭉치 계정화 패턴]].)

### 3.6 생기부 재배선
- 원자료가 아니라 **요약본(요약+첨언)**을 대상으로.
- 그룹핑을 이름이 아니라 **studentNo** 기준으로(동명이인 해결).

## 4. 빌드 순서 (권장)

1. **검증** — WSL `npm run dev`로 현재까지 컴파일·동작 확인(미검증 코드 다수 + 병렬 챗봇 작업과 정합).
2. **명렬 배선** — §3.1 앱↔명렬, §3.2 학생 식별.
3. **요약본·첨언·학생열람** — §3.3~3.5.
4. **생기부 재배선** — §3.6 (studentNo 기준, 요약본 종합).
5. **운영 안전판** — 무료 한도 캡 + 비용 미터(어뷰징·과금).
6. **주권 모드 트랙** — §2 (Apps Script 생성기). 별도 스택, 중앙 안정화 후.

## 5. 백로그
